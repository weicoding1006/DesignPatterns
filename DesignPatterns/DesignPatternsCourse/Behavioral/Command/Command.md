# 命令模式 (Command Pattern)

## 📖 模式介紹 (Introduction)

命令模式是一種行為型設計模式，其主要精神是**「將請求（操作）封裝為一個獨立的物件」**。
在沒有使用命令模式的情況下，呼叫者直接呼叫接收者，導致兩者高度耦合。透過引入「命令物件」，我們能在兩者之間建立緩衝，不但能降低系統耦合度，還能支援如「延遲執行」、「狀態復原 (Undo)」和「重做 (Redo)」等強大功能。

## 🧩 類別角色與對應程式碼 (Core Components)

我幫你看過目前的實作，**程式碼完全正確**！
各個類別完美結合並對應了命令模式中的主要角色：

1. **Command (命令介面)** - `ICommand.cs`
   - 定義了統一的操作介面，宣告了 `Execute()` 與 `Undo()` 方法。
2. **ConcreteCommand (具體命令)** - `ConcreteCommand.cs` 中的 `LightOnCommand` 與 `LightOffCommand`
   - 實作了 `ICommand` 介面，內部保存了真正的被操作者（接收者 `Light`）。
   - 在 `Execute()` 與 `Undo()` 中，實際呼叫呼叫了接收者的功能與其反向邏輯。
3. **Receiver (接收者)** - `Light.cs`
   - 真正執行業務邏輯、完成請求的物件。任何類別都能夠成為接收者，只要它能完成實際的工作。
4. **Invoker (呼叫者/發起者)** - `RemoteControl.cs`
   - 負責觸發命令（例如遙控器按鈕），它不會知道系統是如何執行的，只負責呼叫命令物件的 `Execute()`。
   - 內部透過一個 `Stack<ICommand>` 成功保存了命令的執行歷史紀錄，並完美實作了 `PressUndoButton()` 的復原功能。
5. **Client (客戶端)**
   - 通常存在於主程式中（如 `Program.cs`），負責將接收者、命令物件與呼叫者串接起來，發起實際的操作流程。

## 💡 常用情境 (Common Use Cases)

命令模式在許多軟體系統中都有廣泛應用，以下是幾個最常見的情境：

1. **支援復原 (Undo) 與重做 (Redo) 的軟體系統**
   - **文字編輯器 / Office 軟體**：每一次的打字、刪除、貼上等操作都被包裝成 Command 並放進歷史堆疊 (Stack) 裡。當使用者按下 `Ctrl + Z` 時，系統只要從堆疊將最新的命令取出並呼叫 `Undo()`。
   - **繪圖與影像處理 (Photoshop)**：套用濾鏡、裁切、筆刷等功能也是經典的 Command 應用。

2. **工作佇列與背景排程 (Job Queues & Scheduling)**
   - 將系統即將要執行的長期任務或背景腳本包裝成 Command，加入執行佇列（Queue）中。此時可以從容控制執行序的排程，例如隨時增加、取消或是暫停命令。
   - 例如：報表產生的非同步背景作業、資料庫的批次寫入任務。

3. **使用者介面按鈕與選單 (GUI Buttons and Menus)**
   - 將應用程式中不同視窗的按鈕事件分開定義成不同的獨立 Command 類別，有助於降低 UI 畫面（Invoker）和業務邏輯層（Receiver）之間的耦合度。每個按鈕功能雖不相同，但對於按鈕類別來說其實都只需要呼叫一個 `Command.Execute()`。

4. **支援巨集功能 (Macro Commands)**
   - 一個「巨集命令」裡面可以包含一整個 `List<ICommand>`。當執行巨集時，它就循序跑過陣列裡面所有子命令的 `Execute()` 方法。
   - 例如：智慧家庭控制系統裡面的「開啟劇院模式」：會同時送出命令（調暗燈光 + 關閉窗簾 + 開啟電視 + 開啟音響）。
