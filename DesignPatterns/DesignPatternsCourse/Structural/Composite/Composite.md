# Composite 模式 (組合模式)

組合模式 (Composite Pattern) 是一種**結構型 (Structural)** 設計模式。它允許你將物件組合成「樹狀結構」來表示「部分—整體」的階層關係。

組合模式最核心的價值在於：**讓客戶端 (Client) 能夠以一致的方式處理「個別物件 (獨立的葉節點)」與「組合物件 (包含子節點的樹枝)」**，而不需要寫額外的程式碼來分辨它們。

## 1. 核心角色 (Components)

1. **Component (抽象構件)**：
   * 為樹狀結構中的所有物件宣告統一的介面。
   * 定義了所有具體類別共用的行為（例如：`Operation()`）。
   * （可選）宣告管理子節點的方法，如 `Add()`, `Remove()`, `GetChild()`。
2. **Leaf (葉節點)**：
   * 代表樹狀結構中的末端物件，它**沒有**子節點。
   * 實作 Component 介面中定義的實際行為。
3. **Composite (組合節點 / 樹枝節點)**：
   * 包含子節點 (可以是 Leaf，也可以是其他的 Composite)。
   * 實作 Component 介面，通常其行為是**將操作委派 (Delegate) 給其下的所有子節點**。
   * 實作管理子節點的方法（新增、移除）。
4. **Client (客戶端)**：
   * 透過 Component 介面與樹狀結構中的物件互動。

## 2. 適用場景

當你的系統中存在**「整體與部分」**的階層關係（樹狀結構），且你希望**忽略個體與組合的差異**，統一對待它們時。
* **檔案系統**：資料夾內可以有檔案或其他資料夾。不論是計算大小或列印名稱，對檔案和資料夾的操作概念是一致的。
* **GUI 視窗層次**：一個 Window 裡面有 Panel，Panel 裡面有 Button 和 TextBox。
* **組織架構**：總公司下有分公司和部門，部門下有員工。
* **電商購物車**：商品清單中可能包含單一商品，或是包含多個商品的「促銷組合包」。

## 3. 透明方式 (Transparency) vs 安全方式 (Safety)

* **透明方式 (Transparency)**：
  把管理子節點的方法 (如 `Add`, `Remove`) 宣告在 **Component** 中。
  * **優點**：客戶端可以將 Leaf 和 Composite 視為一致，達成「透明化」。
  * **缺點**：Leaf 節點其實不該具備這些方法，呼叫時可能會拋異常 (例外)。
* **安全方式 (Safety)**：
  把管理子節點的方法保留在 **Composite** 中。
  * **優點**：架構較為安全，Leaf 類別不會實作無意義的行為。
  * **缺點**：客戶端在操作時，必須先判斷該物件是不是 Composite 型別。

## 4. 優缺點比較

**優點**：
1. **客戶端程式碼簡單**：統一處理組合結構和單一物件。
2. **易於擴展**：可輕易加入新的 Leaf 或 Composite 類別，符合「開閉原則」。

**缺點**：
1. **設計過度一般化**：所有的宣告都在抽象層裡，若想要限制組合中的元件類型會比較困難，必須在執行時期手動檢查。
