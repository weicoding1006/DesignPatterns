# 創建型設計模式 (Creational Design Patterns)

## 1. 什麼是創建型模式？

創建型設計模式 (Creational Patterns) 專注於**「物件的建立方式」**。

它們以特定的方式建立物件，以提高程式碼的靈活性和可重用性。設計此類模式的核心問題是：

> **「誰來建立物件？在哪裡建立？如何建立？」**

在大型系統中，若物件的建立散落在程式碼各處（即到處都是 `new SomeClass()`），一旦需要更換實作類別或調整初始化參數，就可能需要修改大量地方，造成脆弱且難以維護的耦合。

**創建型模式將物件的建立邏輯封裝起來**，讓系統不依賴於「具體類別是如何被建立、組合和表示的」，進而達到解耦的目的。

---

## 2. GoF 的五種創建型模式

GoF（Gang of Four，四人幫）共定義了以下五種創建型設計模式：

| 模式名稱 | 一句話說明 | 適用情境 |
|:---|:---|:---|
| **Singleton（單例）** | 確保一個類別只有一個實例，並提供全域存取點 | 共享資源管理（設定檔、Logger、DB 連線池） |
| **Factory Method（工廠方法）** | 讓子類別決定要建立哪一種物件 | 框架定義流程、子類別決定具體產品 |
| **Abstract Factory（抽象工廠）** | 提供一個介面以建立「一系列相關物件」，無需指定具體類別 | 需建立同一主題的多個相關物件（如 UI 組件家族） |
| **Builder（建造者）** | 將複雜物件的建構過程與表示方式分離 | 物件有大量可選參數或多步驟組裝流程 |
| **Prototype（原型）** | 透過複製（Clone）現有物件來建立新物件 | 物件建立成本高，且新物件與現有物件差異小 |

---

## 3. 各模式簡介

### 3.1 Singleton（單例模式）

**核心概念：** 在應用程式的生命週期中，某個類別**只存在唯一一個實例**。

**解決的問題：** 防止多個地方重複建立消耗資源的物件（例如資料庫連線、設定讀取器），確保大家共用同一份狀態。

**現實類比：** 一間公司只有一個執行長（CEO），無論從哪個部門發請示，最終批示的都是同一個人。

**經典場景：**
- `DbContext` / 資料庫連線池管理
- 應用程式設定（`AppSettings`）
- 日誌記錄器（`Logger`）

**設計要點：**
- 建構子設為 `private`，防止外部直接 `new`
- 提供靜態存取點（`Instance` 屬性）
- 多執行緒環境需考慮 **執行緒安全（Thread-safe）** 問題

---

### 3.2 Factory Method（工廠方法模式）

**核心概念：** 在父類別中定義一個「建立物件」的介面（工廠方法），但**將實際建立哪種物件的決定權交給子類別**。

**解決的問題：** 當你無法預先知道要建立哪種具體類別，或希望子類別能夠客製化所建立的物件類型時使用。

**現實類比：** 加盟速食店（父類別）定義了「製作漢堡」的標準流程，但各地分店（子類別）可以決定在地口味的食材內容。

**與簡單工廠的差異：**
> 簡單工廠（Simple Factory）只是將 `new` 集中在一個地方，並非正式的 GoF 設計模式；Factory Method 則透過繼承多型讓子類別掌控生產邏輯，符合開閉原則（OCP）。

**設計要點：**
- 父類別包含使用產品的高層邏輯
- 子類別覆寫 (override) 工廠方法以回傳具體產品
- 產品必須實作共同介面，父類別才能統一使用

---

### 3.3 Abstract Factory（抽象工廠模式）
cd
**核心概念：** 提供一個介面，讓呼叫方可以建立**一個系列（家族）的相關物件**，而毋需指定具體類別。

**解決的問題：** 當系統需要與多組相關產品家族合作，且你希望確保同一家族的物件會一起被使用（不會出現「Windows 按鈕」配「Mac 選單」這種奇怪組合）。

**現實類比：** 一間傢俱工廠（抽象工廠）可以生產同系列的「椅子」、「桌子」、「沙發」；你選擇「北歐風工廠」，就會得到一整套北歐風傢俱，不會摻雜其他風格。

**與 Factory Method 的差異：**
> Factory Method 著重在「建立**單一**產品」的繼承機制；Abstract Factory 著重在「建立**一系列相關**產品」的介面設計。

**設計要點：**
- 定義一個抽象工廠介面，宣告各種產品的建立方法
- 每個具體工廠類別對應一個產品家族
- 呼叫方只依賴抽象介面，不感知具體工廠或產品

---

### 3.4 Builder（建造者模式）

**核心概念：** 將複雜物件的**建構過程**（步驟與順序）與物件的**最終表示方式**（成品）分離開來。

**解決的問題：** 當一個物件有大量的可選欄位或多個建構步驟，使用「巨型建構子」（Telescoping Constructor）會讓程式碼難以閱讀和維護。

**現實類比：** 建造一棟房子，你可以選擇要不要游泳池、要幾層樓、用什麼建材。建築團隊（Builder）負責執行每一個步驟，而你（Director）負責指揮先後順序。

**設計要點：**
- **Builder**：宣告建構各部件的介面（如 `BuildWalls()`, `BuildRoof()`）
- **ConcreteBuilder**：實作具體建構步驟，並提供 `GetResult()` 取得成品
- **Director（可選）**：封裝建構步驟的順序與組合，對呼叫方隱藏細節
- **Product**：最終被建造出的複雜物件

> **C# 的 Fluent Builder** 是一種常見的簡化版變體（如 `WebApplication.CreateBuilder(args).Build()`），省略 Director 改為呼叫方自行串聯方法。

---

### 3.5 Prototype（原型模式）

**核心概念：** 透過**複製（Clone）**一個現有的物件（稱為「原型」），來建立新的物件，而不是重新用 `new` 從頭建立。

**解決的問題：** 當物件建立成本很高（例如需要讀取大量設定、連線外部服務），或是新物件與現有物件的差異很小時，Clone 更有效率。

**現實類比：** 複印機（Prototype）不需要重新排版文件，直接複印現有文件，然後你再手動修改幾個字。

**淺拷貝 vs 深拷貝：**
| | 淺拷貝 (Shallow Copy) | 深拷貝 (Deep Copy) |
|:---|:---|:---|
| 說明 | 複製物件本身，內部的參考型別欄位仍指向同一記憶體 | 完整複製所有層級的資料，內部物件也是全新的 |
| 風險 | 修改 Clone 的內部物件，會影響到原始物件 | 無上述風險，但實作較複雜 |
| C# 方法 | `MemberwiseClone()` | 手動實作或使用序列化 |

**設計要點：**
- 讓產品類別實作 `ICloneable` 介面（或自訂 `Clone()` 方法）
- 決定你的系統需要「淺拷貝」還是「深拷貝」

## 4. 五大模式比較

| 模式 | 建立單位 | 機制 | 主要意圖 |
|:---|:---|:---|:---|
| Singleton | 單一物件（僅一份） | 靜態屬性 | 控制實例數量為 1 |
| Factory Method | 單一產品 | 繼承、覆寫 | 讓子類別決定具體產品類別 |
| Abstract Factory | 多個相關產品（一家族） | 組合、介面 | 確保產品家族的一致性 |
| Builder | 單一複雜物件 | 步驟拆解 | 分離建構過程與表示方式 |
| Prototype | 新物件（從現有物件克隆） | Clone | 降低物件建立成本 |

## 5. 在 ASP.NET Core Web API 的日常應用

| 創建型模式 | 對應的 ASP.NET Core 場景 |
|:---|:---|
| **Singleton** | `services.AddSingleton<T>()` ── 整個應用週期共用一個實例 |
| **Factory Method** | `IHttpClientFactory` ── 框架決定如何建立 `HttpClient` |
| **Abstract Factory** | EF Core 的 `DbContextOptions` ── 切換 SQL Server / SQLite 毋須改業務程式碼 |
| **Builder** | `WebApplicationBuilder`（`builder.Build()`）; `ContainerBuilder`（Autofac） |
| **Prototype** | `IOptions<T>` 的快照機制（`IOptionsSnapshot`）; 物件的 DeepCopy 快取策略 |

## 6. 如何選擇合適的模式？

```
物件建立的問題
│
├─ 這個類別應該只有一個實例？
│   └─➜ Singleton
│
├─ 建立哪種物件的決定需要延遲到子類別？
│   └─➜ Factory Method
│
├─ 需要同時建立一組相關的物件，確保它們來自同一家族？
│   └─➜ Abstract Factory
│
├─ 物件有複雜的建構步驟或大量可選參數？
│   └─➜ Builder
│
└─ 建立物件成本高，需要快速複製現有物件？
    └─➜ Prototype
```

---

## 7. 小結

創建型模式的本質，都是在回答同一個問題：
**「如何讓物件的建立更靈活、更可控、更易於擴展？」**

良好的創建型設計能讓你的系統：
- **對修改封閉**：換掉底層實作，上層程式碼不需要改動
- **對擴展開放**：增加新的產品，不需要修改現有工廠或建構流程
- **降低耦合**：呼叫方不感知具體類別，只依賴抽象介面
